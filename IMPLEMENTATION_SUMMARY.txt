ARCHITECTURE AT A GLANCE
================================================================================

REQUEST FLOW:
  User Request
    ↓
  [DRF ViewSet + DocumentRules Permission Class]
    ↓
  [Extract operation type: read/write/delete]
    ↓
  [Load SecurityPolicy rules for (project, collection, operation)]
    ↓
  [RuleEngine.check(): Evaluate condition tree against request+document]
    ↓
  [Condition types: auth_check, field_check, owner_check, role_check]
    ↓
  [Log to PolicyAuditLog]
    ↓
  [Return: Allowed (200) or Denied (403)]

DATABASE:
  core_project (existing)
    ├── id: UUID
    ├── name, slug, owner, api_key
    └── ...

  core_document (new)
    ├── id: UUID
    ├── project_id (ForeignKey)
    ├── collection: "documents", "posts", etc.
    ├── data: JSONField (arbitrary document)
    ├── owner: ForeignKey(User)
    ├── created_by, updated_by: ForeignKey(User)
    └── created_at, updated_at

  rules_security_policy (new)
    ├── id: UUID
    ├── project_id (ForeignKey)
    ├── collection: "documents"
    ├── rule_type: "read" | "write" | "delete"
    ├── condition_json: JSONField (condition tree)
    ├── active: Boolean
    ├── priority: Integer (evaluation order)
    ├── description: TextField
    └── created_at, updated_at

  rules_policy_audit_log (new)
    ├── id: UUID
    ├── project_id (ForeignKey)
    ├── user_id (ForeignKey)
    ├── collection, operation, document_id
    ├── allowed: Boolean
    ├── reason: TextField (why allowed/denied)
    └── created_at (indexed)

================================================================================
CONDITION JSON FORMAT
================================================================================

Structure:
{
  "operator": "and" | "or",
  "conditions": [
    {
      "type": "auth_check" | "field_check" | "owner_check" | "role_check",
      "value": { /* type-specific */ }
    },
    ...
  ]
}

Condition Types:

1. auth_check — Check if authenticated
   {"type": "auth_check", "value": {"op": "!=" | "==", "rhs": "null"}}

2. field_check — Compare document fields
   {"type": "field_check", "value": {
     "path": "data.owner" | "data.is_public",
     "op": "==" | "!=" | "<" | "<=" | ">" | ">=" | "in" | "contains" | "matches",
     "rhs": "literal_value" | null,
     "rhs_field": "request.auth.uid" | null
   }}

3. owner_check — Check document ownership
   {"type": "owner_check", "value": {"field": "owner_id"}}

4. role_check — Check user project role (Phase 2)
   {"type": "role_check", "value": {"role": "admin" | "editor" | "viewer"}}

Examples:

  Owner Only:
  {
    "operator": "and",
    "conditions": [
      {"type": "auth_check", "value": {"op": "!=", "rhs": "null"}},
      {"type": "owner_check", "value": {"field": "owner_id"}}
    ]
  }

  Public or Owner:
  {
    "operator": "or",
    "conditions": [
      {"type": "field_check", "value": {"path": "data.is_public", "op": "==", "rhs": "true"}},
      {"type": "owner_check", "value": {"field": "owner_id"}}
    ]
  }

  Field Reference:
  {
    "operator": "and",
    "conditions": [
      {"type": "auth_check", "value": {"op": "!=", "rhs": "null"}},
      {"type": "field_check", "value": {
        "path": "data.owner",
        "op": "==",
        "rhs_field": "request.auth.uid"
      }}
    ]
  }

API USAGE EXAMPLES
================================================================================

1. CREATE DOCUMENT (Authenticated User)

  POST /api/v1/documents/
  Authorization: Bearer <token>
  Content-Type: application/json

  {
    "project": "<project-uuid>",
    "collection": "documents",
    "data": {"title": "My Document", "content": "..."}
  }

  Response:
  {
    "id": "<doc-uuid>",
    "project": "<project-uuid>",
    "collection": "documents",
    "data": {"title": "My Document", "content": "..."},
    "owner": {"id": 1, "email": "alice@example.com", ...},
    "created_by": {...},
    "created_at": "2026-06-18T...",
    "updated_at": "2026-06-18T..."
  }

2. READ DOCUMENT (Authenticated User)

  GET /api/v1/documents/<doc-uuid>/
  Authorization: Bearer <token>

  Response: 200 OK if user can read (authenticated)
  Response: 403 Forbidden if rule denies

3. UPDATE DOCUMENT (Owner Only)

  PATCH /api/v1/documents/<doc-uuid>/
  Authorization: Bearer <token>
  Content-Type: application/json

  {"data": {"title": "Updated Title"}}

  Response: 200 OK if user is owner
  Response: 403 Forbidden if user is not owner (rule check fails)

4. DELETE DOCUMENT (Owner Only)

  DELETE /api/v1/documents/<doc-uuid>/
  Authorization: Bearer <token>

  Response: 204 No Content if user is owner
  Response: 403 Forbidden if user is not owner

DESIGN DECISIONS
================================================================================

1. JSON Condition Trees (vs Text DSL)
   ✅ Structured format for Phase 1
   ✅ Optional text parser in Phase 2
   ✅ Type-safe evaluation
   ✅ Easy to validate & store

2. Application-Level Evaluation (vs Database-Only)
   ✅ More flexible (custom logic, cross-service)
   ✅ Easier to debug (audit logs, testing)
   ✅ Optional Postgres RLS for hardening
   ✅ Language-native Python expressions

3. Fail-Safe Default (vs Fail-Open)
   ✅ No matching rule = DENY
   ✅ Explicit allow required
   ✅ Prevents accidental exposure

4. Priority-Based Ordering
   ✅ Clear precedence (no ambiguity)
   ✅ First match wins
   ✅ Performance: early exit possible
   ✅ Familiar: like middleware/permission chains

================================================================================
TROUBLESHOOTING
================================================================================

Issue: Rule not being enforced
  → Check if rule active=True
  → Verify DocumentRules in permission_classes
  → Check PolicyAuditLog for evaluation
  → Verify rule priority (higher = first)
  → Test in Django shell with manual evaluation

Issue: 403 Forbidden but should be allowed
  → Check audit logs for actual rule that denied
  → Review condition_json structure
  → Verify document fields match condition paths
  → Check is_staff flag if using admin bypass

Issue: Audit log too large
  → Create management command to prune old logs
  → Add to Celery beat for automatic cleanup
  → Use DocumentRulesNoAudit for high-volume scenarios

Issue: Postgres RLS policy not applied
  → RLS is optional; app-level enforcement is primary
  → Call enable_rls_on_table() to enable
  → Check Postgres logs for SQL errors
  → Complex rules may not translate to SQL

================================================================================
PERFORMANCE CHARACTERISTICS
================================================================================

Rule Evaluation:
  Typical: 1-5ms per request
  With 10 rules: ~10-50ms
  With 100 rules: ~100-500ms (rare)

Optimization opportunities:
  ✅ Cache frequently evaluated rules (Phase 2)
  ✅ Compile rules to faster bytecode (Phase 2)
  ✅ Use Postgres RLS for database-level enforcement (optional)
  ✅ Batch evaluations for high-volume scenarios
  ✅ Use DocumentRulesNoAudit to skip audit logging

Audit Logging:
  ✅ Every evaluation logged to PolicyAuditLog
  ✅ Lightweight: ~50 bytes per log entry
  ✅ Can be pruned after retention period
  ✅ Queryable for debugging & analytics
