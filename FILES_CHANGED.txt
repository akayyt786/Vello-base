KEY IMPLEMENTATION DETAILS:
===========================

Authentication Flow:
  Client → Register/Login → DRF ViewSet → CustomJWTAuthentication
         → Verify Signature → Check Blacklist → Attach Claims → Return User

Token Structure:
  Access Token (15 min):
    - user_id, email, email_verified, sign_in_provider
    - Custom claims from UserProfile.custom_claims
    - Firebase-like metadata structure
  
  Refresh Token (7 days):
    - Standard JWT with jti claim for tracking
    - Checked against RefreshTokenBlacklist on use

Custom Claims Flow:
  Admin calls /api/auth/set-custom-claims/ with user_id + claims JSON
  → Validated against 1000-byte limit (Firebase-compatible)
  → Stored in UserProfile.custom_claims
  → Automatically injected into next access token by CustomTokenObtainPairSerializer
  → Claims appear immediately on next login (no propagation delay)

Token Blacklist Flow:
  User calls /api/auth/logout/ with refresh token
  → Token decoded to extract jti
  → RefreshTokenBlacklist entry created with jti + expiration
  → CustomJWTAuthentication checks blacklist on every request
  → Blacklisted tokens raise 401 Unauthorized

Multi-Tenant Support:
  User provides optional project_id during login
  → project_id added to JWT claims
  → MultiTenantMiddleware extracts project_id from token
  → Used for RLS policies and permission checks
  → Enables multiple projects per user with different roles

INTEGRATION POINTS:
===================

For Client Applications (Web/Mobile):

1. On app startup:
   - Check for valid access token in memory/session
   - If expired, call /api/auth/refresh/ with stored refresh token
   - If both expired, redirect to /api/auth/login/

2. On login:
   - Call /api/auth/login/ with email + password
   - Store access token (memory/sessionStorage)
   - Store refresh token (httpOnly cookie or secure storage)

3. On every API request:
   - Include Authorization header: Bearer {access_token}
   - If 401 response, refresh token and retry
   - If refresh fails, redirect to login

4. On logout:
   - Call /api/auth/logout/ with refresh token
   - Clear stored tokens
   - Redirect to login page

5. Getting user profile:
   - Call /api/auth/me/ with Bearer token
   - Returns email_verified, sign_in_provider, custom_claims

For Backend Integrations:

1. Accessing custom claims in views:
   - Use: request.user to get User object
   - Use: request.token_claims.get('role') to access custom claims
   - Use: request.token_claims.get('admin') for permission checks

2. Verifying tokens:
   - CustomJWTAuthentication automatically verifies in DRF views
   - For non-DRF endpoints, use: from rest_framework_simplejwt.tokens import Token

3. Checking if user is admin:
   - Check: request.user.is_staff (Django staff flag)
   - Check: request.token_claims.get('admin') (custom claim)

================================================================================

TROUBLESHOOTING:
================

Token Expired / 401 Unauthorized:
  1. Check if token expiration (iat + lifetime) has passed
  2. Call /api/auth/refresh/ with refresh token to get new access token
  3. If refresh token also expired (> 7 days), require user to login again

Token Blacklisted After Logout:
  1. User cannot reuse old refresh token
  2. If user tries, /api/auth/refresh/ returns 401
  3. Redirect to login page

Invalid Custom Claims:
  1. Check JSON is valid (use JSONLint)
  2. Check total size < 1000 bytes: echo -n 'JSON' | wc -c
  3. Check user exists: call /api/auth/me/ to verify

Database Errors:
  1. Ensure migrations ran: python manage.py showmigrations
  2. Ensure PostgreSQL is running
  3. Check DATABASE_NAME, DATABASE_USER, DATABASE_PASSWORD in .env

JWT Signing Errors:
  1. Ensure DJANGO_SECRET_KEY is set in .env
  2. Never change SECRET_KEY in production (all tokens become invalid)
  3. If rotated, all users must login again
